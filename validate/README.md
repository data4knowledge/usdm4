# validate/

Tools and reference data for assessing the two USDM4 validation engines ‚Äî
`USDM4.validate(...)` (the d4k Python rule library) and `USDM4.validate_core(...)`
(the CDISC CORE engine wrapper) ‚Äî and for cross-checking them against each
other.

## Standard test set

`samples/sample_usdm_1.json` through `samples/sample_usdm_6.json` are the
canonical test inputs. They cover a mix of structural shapes the two
engines should agree on. Use these for day-to-day regression checks
after rule library changes.

The full regression sweep is `run.sh` over every sample:

```bash
for f in validate/samples/sample_usdm_*.json; do
    ./validate/run.sh "$f"
done
```

Outputs land alongside each input as `<stem>_core.yaml`, `<stem>_d4k.yaml`,
`<stem>_vs_d4k.yaml`, and `<stem>_vs_d4k.txt`. All four are gitignored ‚Äî
re-run the loop to regenerate.

## CLIs

All CLIs are invoked from the repo root.

### `core.py` ‚Äî CORE engine

```bash
python validate/core.py samples/sample_usdm_1.json [-o out.yaml] [--cache-dir /path/to/cache]
```

Output is a structured YAML file with findings grouped by rule, each error
showing entity, instance_id, path, and relevant details. Execution errors
(rules that don't apply to a given entity type) are counted but excluded
from findings.

### `d4k.py` ‚Äî d4k engine

```bash
python validate/d4k.py samples/sample_usdm_1.json [-o out.yaml]
```

Runs the usdm4 Python rule library (`USDM4.validate`). YAML emits every
rule's outcome (Success / Failure / Exception / Not Implemented), so it
can be diffed against the CORE YAML.

### `compare.py` ‚Äî alignment

```bash
python validate/compare.py <core.yaml> <d4k.yaml> [-o alignment.yaml] [--text] \
    [--core-rules-json /path/to/rules/usdm/4-0.json]
```

Produces a rule-by-rule alignment report. Per-rule classifications:
`aligned_pass`, `aligned_fail`, `count_mismatch`, `d4k_only`, `core_only`,
`d4k_exception`, `d4k_not_impl`, `core_only_rule`. The CORE YAML lists only
failing rules, so "CORE passed" and "CORE did not run this rule" collapse
to `Pass-or-NA` at the rule-id level.

`--core-rules-json` overrides the default path to the CDISC CORE rules JSON
cache (default: `~/Library/Caches/usdm4/core/rules/usdm/4-0.json`); used to
reconcile CORE-NNNNNN ids to DDF000NN ids.

### `run.sh` ‚Äî one-shot wrapper

```bash
./validate/run.sh samples/sample_usdm_1.json [output_dir]
```

Runs all three steps end-to-end on a single JSON file. Honours `PYTHON`
(interpreter) and `CACHE_DIR` (passed to `core.py`) environment overrides.
Outputs land alongside the input by default, or in `output_dir` if given.

The four sample-output filenames produced are:

```
<stem>_core.yaml      CORE engine result
<stem>_d4k.yaml       d4k engine result
<stem>_vs_d4k.yaml    rule-by-rule alignment (machine-readable)
<stem>_vs_d4k.txt     rule-by-rule alignment (human-readable)
```

These are gitignored ‚Äî they are regenerated by re-running `run.sh`.

## Corpus baseline (`corpus_cre_0_16/`)

A frozen reference: the CRE-vs-d4k alignment across 234 protocols, captured
under CRE 0.16.0 with the rule library at 100% V4 coverage. **Read-only.**

Top-level summary artefacts:

- `engine_diff.md` ‚Äî human-readable rule-by-rule classification across the
  full corpus (aligned, under_reporting, over_reporting buckets).
- `engine_diff.yaml` ‚Äî same content, machine-readable.
- `manifest.yaml` ‚Äî what ran on what.

Per-protocol evidence files (`<id>_core.yaml`, `<id>_d4k.yaml`,
`<id>_vs_d4k.yaml`, `<id>_vs_d4k.txt`) are committed for traceability ‚Äî
when `docs/cre_issues.md` claims a CRE issue affects a specific rule on
specific files, the per-file YAMLs are the receipts.

`logs/` is the runtime log output of the original corpus run; gitignored
because it's operational ephemera, not evidence.

The corpus runner that produced this baseline (`corpus.py`, `corpus.sh`,
`engine_diff.py`) was retired on 2026-05-02. If a future systematic re-run
is ever needed, those scripts can be retrieved from git history.

`eval_corpus.py` and `corpus_adapter.py` live alongside the validation CLIs
but belong to a different workflow ‚Äî they're the assembler eval harness,
not the validation-engine wrapper. `eval_corpus.py` walks a corpus,
assembles each protocol's `ground_truth.yaml` into USDM JSON via
`Assembler.execute`, and optionally runs the validation engines on the
result. `corpus_adapter.py` bridges shape divergences between the corpus's
`ground_truth` content and `AssemblerInput`. See
`docs/assembler_validation_findings.md` and `docs/corpus_extractor_fixes.md`
for the divergence catalogue and per-transform rationale.

`eval_corpus.py`'s default `--output-dir` is `validate/eval_output/`
(anchored to the script's location, so the path is the same regardless of
the cwd you launch it from). That directory is gitignored ‚Äî it holds
working scratch (per-protocol YAMLs, summary, engine diff) that's
regenerated by re-running the harness.
